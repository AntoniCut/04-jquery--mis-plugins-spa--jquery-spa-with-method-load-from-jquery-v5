Plugin jQuery para SPAs con método .load() — v5

jquery-spa-with-method-load-from-jquery es un plugin jQuery diseñado para crear aplicaciones de una sola página (SPA) sin frameworks. Carga fragmentos HTML dinámicamente mediante el método jQuery.load(), inyectándolos en contenedores del DOM definidos por rutas configuradas en un manifiesto ligero.

La versión 5 extiende el motor SPA con lazy loading de módulos de ruta mediante import(), integración con jQuery UI (draggable, tooltip, themes), soporte nativo para la View Transitions API, reescritura automática de URLs en HTML inyectado, metadatos dinámicos por ruta, navegación con History API, caché de módulos, eventos personalizados y manejo robusto de errores para desacoplar el loader visual de la lógica de navegación.

Características Principales

🔌

Plugin jQuery — $.fn.spaWithMethodLoadFromJQuery

Se registra como método del prototipo jQuery e inicializa la SPA sobre el contenedor raíz (p. ej. $('#layout').spaWithMethodLoadFromJQuery(options)). Devuelve this para encadenamiento. Se exporta envuelto en un módulo ES6 (spaWithMethodLoadFromJQueryPlugins()) para integrarse con el bundler del proyecto.

🚀

Lazy Loading de Rutas

Los módulos de ruta se importan dinámicamente con import() solo cuando el usuario navega a esa ruta. Un manifiesto ligero (routeManifest) registra cada ruta con id, path y file, sin importar módulos en el arranque. Los módulos ya cargados se guardan en un Map interno.

🎬

View Transitions API

Cuando el navegador soporta document.startViewTransition(), el plugin envuelve automáticamente el renderizado de la nueva ruta en una transición nativa, logrando animaciones fluidas entre páginas sin configuración adicional.

📋

Route Manifest

El routeManifest es un array de entradas { id, path, file }. El plugin resuelve la ruta activa comparando window.location.pathname normalizado con cada path, y carga el módulo correspondiente bajo demanda.

🔃

Inyección con jQuery .load()

Cada ruta define un objeto components donde la clave es el selector CSS del contenedor (p. ej. #layoutMain) y el valor es la URL del fragmento HTML. El plugin usa $(selector).load(url) para inyectar cada fragmento de forma secuencial. Si un componente no tiene URL, oculta su contenedor con .hide().

🎨

CSS y JS por Ruta

Cada ruta puede declarar styles (array de { href }) y scripts (array de { src, type?, isModule?, exportFunctionName? }). Los CSS se precargan con <link rel="preload"> antes de aplicarse. Los scripts clásicos usan $.getScript(); los módulos ES6 usan import() dinámico con ejecución opcional de exportFunctionName.

⚡

Caché y Eventos Personalizados

Los módulos de ruta se cachean en un Map interno para evitar importaciones redundantes. El plugin emite eventos spa:route-loaded, spa:first-route-loaded y spa:route-load-error para desacoplar el loader visual de la navegación SPA.

🔗

Reescritura de URLs en HTML

Al inyectar fragmentos HTML, el plugin reescribe automáticamente las rutas relativas de src, href, poster y srcset respecto a la URL del fragmento origen, evitando enlaces rotos cuando los componentes referencian assets en su propio directorio.

🧭

Navegación con History API

Intercepta clics en a[data-id] y a[data-route]. Con data-id resuelve la ruta en el manifest; con data-route importa directamente el módulo por nombre de archivo. Actualiza la URL con history.pushState() guardando routeFile y favicon en el state. El evento popstate restaura la vista usando primero routeFile del historial (más rápido, usa caché).

📐

Normalización de Rutas y Base

Funciones internas normalize() y buildPathname() eliminan la base configurada y los slashes sobrantes para comparar rutas de forma fiable. Evita pushState duplicados cuando la URL solo difiere en la barra final (/base/ vs /base).

🏷️

Metadatos Dinámicos por Ruta

Cada ruta puede definir pageTitle, headerTitle y favicon. El plugin actualiza <title>, #headerTitle / #footerTitle y el favicon al navegar. El favicon solo se reasigna cuando el archivo cambia realmente, evitando parpadeos en navegación atrás/adelante.

🚫

Ruta 404 Integrada

Si la URL no coincide con ninguna entrada del manifest, el plugin carga automáticamente la ruta 404NotFoundPage (también reconoce paths 404, 404-not-found o ids con 404). Funciona en carga inicial, clics internos y popstate. Emite spa:route-load-error si no hay 404 configurada.

🛡️

Manejo de Errores

Los fallos al cargar componentes, importar módulos o renderizar rutas disparan spa:route-load-error con detail.source y detail.message. En la carga inicial también emite spa:first-route-loaded como fallback para no bloquear el loader. Los errores 404 en .load() muestran un mensaje en el contenedor afectado.

⏳

Loader Inicial Desacoplado

El efecto effect-loading-page.js escucha spa:first-route-loaded y spa:route-load-error para ocultar el loader cuando la primera ruta SPA termina (éxito o error), con timeout de fallback de 6 segundos, manteniendo la UX fluida desde el arranque.

🧩

jQuery UI — Carga Bajo Demanda (libs)

Cada ruta puede declarar libs: [{ name: 'tooltip' }, ...]. Tras renderizar el DOM, el plugin invoca settings.libLoader(name) para importar módulos de jQuery UI solo cuando la ruta los necesita (datepicker, dialog, draggable, sortable, etc.), optimizando el peso inicial.

✋

Elementos Draggable

Tras cargar las libs de la ruta, enableDraggables() busca elementos con clase .draggable y aplica .draggable({ scroll: false }) de jQuery UI. La opción global draggable: true en la configuración del plugin habilita esta funcionalidad en el proyecto.

☰

Navbar Interactivo (actionsNavbar)

Inicializa el menú principal (.navbar__container) y el menú de themes (#linksThemesContainer) con animaciones slideUp / slideDown de jQuery. Cierra menús al hacer clic fuera y garantiza que solo uno esté abierto a la vez. Protegido con try/catch si la vista no incluye navbar.

🎭

Themes jQuery UI Dinámicos

changeThemesJQueryUI() enlaza los links del panel de themes (data-theme) para cambiar el href de #theme en tiempo real. Activa la clase .active en el theme seleccionado e inicializa .tooltip() en el contenedor de themes.

📜

Scripts con Verificación HEAD

Antes de cargar un script clásico, el plugin verifica su existencia con $.ajax({ type: 'HEAD' }). Elimina scripts previos marcados con data-page-script="true" al cambiar de ruta. Soporta type: 'module' e isModule: true de forma equivalente.

🔄

Gestión de Estilos por Página

Al navegar, elimina hojas de estilo anteriores con link[data-page-style="true"] e inserta las nuevas de route.styles mediante preload + stylesheet, reduciendo parpadeos (FOUC) durante el cambio de vista.

🔁

Origen de Navegación (source)

loadContent(route, source) distingue 'init', 'click' y 'popstate'. Solo en clics internos ejecuta pushState; en carga inicial usa replaceState externo y en popstate respeta el historial ya actualizado por el navegador.

Integración jQuery UI

⚙️

Opciones del Plugin

Configuración vía $.extend(): routeManifest, routeModulesBase, base, draggable y libLoader. El archivo spa.js del proyecto pasa el manifest, la base del deploy y el cargador de módulos jQuery UI al inicializar el plugin.

💡

Tooltips en Navbar

Las rutas que cargan la lib tooltip pueden usar .tooltip() de jQuery UI en el panel de themes y en elementos del layout, activándose tras la carga bajo demanda del widget correspondiente.

Pipeline de Desarrollo

El proyecto separa el código fuente (src/) del artefacto servible (app/) y del build de producción minificado (dist/). Gulp orquesta la copia, compilación SCSS y minificación; los servidores Express añaden fallback SPA, live reload y soporte PHP en desarrollo.

📁

Gulp — src → app

La tarea copyAll copia componentes, páginas, rutas, plugins, scripts y efectos desde src/ a app/, y compila globals.scss y los SCSS de cada página a CSS con sourcemaps.

👀

Watch en Tiempo Real

gulp dev ejecuta copyAll y observa cambios en src/. Al modificar SCSS, recompila estilos y copia los cambios a app/. Soporta polling vía CHOKIDAR_USEPOLLING para entornos Linux con límites de inotify.

🌐

Dev Server + BrowserSync

pnpm dev lanza Gulp watch y un servidor Express con fallback SPA, archivos estáticos bajo la base del proyecto, soporte php-cgi y proxy BrowserSync con live reload en app/ e index.html (puerto configurable con DEV_SERVER_PORT).

📦

Build de Producción

pnpm build limpia dist/ y app/, regenera app/ desde src/ y minifica HTML, CSS y JS con htmlmin, clean-css y terser hacia dist/, listo para despliegue.

🔍

Preview Server

pnpm preview sirve el contenido de dist/ con Express, fallback SPA y la misma base de rutas que en desarrollo. Permite validar el build minificado antes de publicar (puerto PREVIEW_SERVER_PORT, por defecto 4173).
